# 存取性分層評分方法

> 文件目的：定義 M3 存取性分層所使用的 6 個評估維度與 4 階評分系統。
> 透明化評分標準讓外部評分者可以用同樣準則重新評估——這是工具可被挑戰的基礎。

---

## ⚠ 重要框架警示：高分 ≠ 對原住民族社群友善

**本評分衡量的是「對外部使用者（研究者、NGO、開發者、任何人）的可取用性」，不是「對原住民族社群的友善度」。**

依此設計，一個**最易被殖民式抽取**的資料庫（資料對外完全開放、無治理機制回饋社群）會在此評分中得到高分。這不是設計錯誤——這是刻意保留的分析邊界：

- **M3（本文件所評）** = FAIR（Findable, Accessible, Interoperable, Reusable）的延伸——衡量抽取易度
- **M2（治理落差矩陣）** = CARE、OCAP、TK Labels、在地化條款的對照——衡量社群權利與資料主權

**CARE 原則當初的設計意圖，就是補 FAIR 對集體權利的忽略**。我們的工具沿用這個分裂，以保留兩個獨立分析軸的概念純粹性。

### 如何正確解讀 M3 分數

**永遠與 M2 治理矩陣一起讀**。單看 M3 會誤導——例如 CIP 的 16/18 分（極易被外人取用）必須搭配 M2 檢視（CIP 僅在「在地化集體權利」類別有 2 格 implemented，其餘大面積 none）。CIP 的真實狀況是「**技術上最開放但治理上有限，且沒有回饋社群的機制**」。

視覺化（M4）時必須設計為兩個矩陣互動連結，讓讀者無法僅看其一就下結論。任何單獨呈現 M3 或 M2 的摘要都須附此框架警示。

---

## 為何需要這個評分體系

本維度體系由 2026-04-12 勘查過程中的實際發現催生——國際原住民族資料治理框架（CARE、OCAP、TK Labels）不涵蓋「存取性」的多層次結構；台灣既有的資料治理討論也未具體化。勘查紀錄 [audit-log.md](../audit-log.md) 揭示了六種失敗模式（SPA 空殼、錯誤架構假設、URL 路徑失效、DNS 消失、API 名存實亡、逾時），直接指向「可存取性需要分層描述」這件事。

**核心判斷**：一個平台「有 API」不等於「API 可用」（如 data.gov.tw 的搜尋 API）；「有資料」不等於「資料可被發現」（如 TICD on OSF）；「開放授權」不等於「實際可下載」（如 AILT 的 All Rights Reserved + 無下載）。

---

## 4 階評分系統（0–3 分）

為降低「中間分數膨脹」，採 4 階而非 5 階或連續刻度：

- **0 分**：最嚴重的障礙；實質不可能或需專業技術對抗
- **1 分**：明顯障礙；需特殊知識、工具或程序
- **2 分**：一般技術門檻內可達；需基本技能
- **3 分**：理想狀態；無障礙或最低摩擦

每分級的具體判準依維度不同。

## 六個存取性維度

### 1. 可發現性 Discoverability

**評估問題**：一個陌生使用者（無內部資訊）能否從公開管道找到這個資料源？

| 分數 | 判準 |
|------|------|
| 0 | 需內部知識、學術人脈或偶然線索才能找到 |
| 1 | 有公開連結但主站無導航到此；需透過搜尋引擎或非官方途徑發現 |
| 2 | 主站有連結但需多層點擊；非首頁顯著位置 |
| 3 | 主站首頁可直接到達 |

**邊界案例**：
- URL 大小寫敏感（如 TCMB `/zh-tw/OpenApi` 對 `/zh-tw/openapi` 失敗）視為次級可發現性問題，但不影響主站層次評分
- 前端 SPA 搜尋在瀏覽器能運作但程式化抓取失敗，仍算「可發現」

---

### 2. 技術門檻 Technical Barrier

**評估問題**：取得資料需要什麼技術能力？

| 分數 | 判準 |
|------|------|
| 0 | 實質不可取得原始資料；需爬蟲對抗嵌入格式（如 YouTube）、破解保護或無法突破的技術障礙 |
| 1 | 需撰寫複雜 API 請求、處理動態渲染、模擬瀏覽器操作 |
| 2 | 基本 HTTP GET 或 API Key 驗證後可取得 |
| 3 | 一般瀏覽器直接點擊下載 |

**邊界案例**：
- ASP.NET postback 等動態介面需 Playwright 模擬才能觸發——視為 1 分
- 需固定 IP 的 API：技術本身簡單但部署複雜度高——歸 1 分

---

### 3. 授權門檻 Authorization Barrier

**評估問題**：取得資料需要什麼授權／認證程序？

| 分數 | 判準 |
|------|------|
| 0 | 需專案核准、機構背書、固定 IP 等高門檻程序 |
| 1 | 需註冊帳號或提交申請（無需核准） |
| 2 | 需同意條款或被動接受使用規範 |
| 3 | 無任何門檻；直接可取得 |

**邊界案例**：
- 平台有混合授權（部分免申請、部分需申請）：依使用者想「完整取用」的門檻評分，較嚴為準
- 預設著作權保留（All Rights Reserved）意味所有再利用需主動取得授權但無明確流程——歸 2 分

---

### 4. 授權清晰度 License Clarity

**評估問題**：使用者能多容易理解並判斷再利用條件？

| 分數 | 判準 |
|------|------|
| 0 | 無任何授權聲明 |
| 1 | 授權條款隱晦或文字不明確（如只有 footer 的「All Rights Reserved」無使用條款頁） |
| 2 | 有明確授權但逐筆可能不同，需逐件判斷 |
| 3 | 全站統一、明確、有獨立使用規範頁的開放授權 |

**邊界案例**：
- 「All Rights Reserved」雖然形式明確但缺乏使用指引——歸 1 分而非 2 分
- 獨立制定的使用規範（如 CIP 使用規範 1.0）和採國際標準（如 CC）在清晰度上同等——都是 3 分

---

### 5. 機構穩定性 Institutional Stability

**評估問題**：這個資料源能多可靠地長期維運？

| 分數 | 判準 |
|------|------|
| 0 | 已消失或失靈（域名不可解析、API 停止運作、網站無法存取） |
| 1 | 近 2 年有路徑斷裂、功能失效、機構轉移或版本不相容 |
| 2 | 運作中但無明確法定義務或長期承諾 |
| 3 | 有明確法定義務維運（法規設立、預算編列、機關職責明文） |

**邊界案例**：
- 遷移到其他平台（如 TICD 從 ianthro.tw → OSF）：視為路徑斷裂 1 分
- 功能性路徑斷裂（如 data.gov.tw 搜尋 API 失靈）：視為路徑斷裂 1 分，不是完全失靈 0 分
- 改版後舊路徑失效但新路徑存在（如 TCMB 2.0）：路徑斷裂 1 分

---

### 6. 可再利用性 Reusability

**評估問題**：取得資料後，能多廣泛地再利用？

| 分數 | 判準 |
|------|------|
| 0 | 禁止再製（預設著作權保留、僅限公開瀏覽） |
| 1 | 僅限學術引用或嚴格限定用途 |
| 2 | 可再利用但有重大限制（禁商業、禁改作、須同授權分享等） |
| 3 | 可完全再利用（含商業）；顯名聲明外無限制 |

**邊界案例**：
- 混合授權來源（如 TCMB）：評估**典型／常見**授權而非最嚴或最鬆；必要時在 rationale 說明分布
- GPL 3.0 的 copyleft 要求衍生作品同樣 GPL——這是條件而非限制，歸 3 分

## 評分流程

1. **先讀來源的 YAML 與 audit-log 對應段落**——每個分數須基於實測證據
2. **逐維度評分**——6 個維度分別評，不綜合判斷
3. **所有分數必填 rationale**（不只 partial/implemented——因為分數是 0-3 連續軸，每個分數都承載判斷）
4. **所有分數必填 evidence_refs**——引用 audit-log、source YAML 或 inventory 的具體行範圍
5. **極端值需反例質疑**：給 0 或 3 前問自己「是否能舉出相反證據？」若能就降／升一級

---

## 跨來源總分與排序

6 維度 × 3 分滿分 = 每來源滿分 18 分。

總分提供整體比較指標，但**單一維度的訊息量可能比總分更重要**。例如：AILT 與 TCMB 可能總分相近但分數分布完全不同——AILT 技術門檻 0（無下載）+ 授權清晰度 1（無條款頁），TCMB 則是技術門檻 1（需固定 IP）+ 授權清晰度 2（10 種標示逐筆不同）。總分相似但治理意涵截然不同。

視覺化（M4）時建議同時呈現總分與維度分布。

## 評分者復現測試

本專案第一輪評分見 [data/access-tiers.yaml](../data/access-tiers.yaml)。

**復現測試標準**：外部評分者依本文件獨立評分，分數差異應在每個維度 ±1 以內。若差異 ≥2，表示本文件判準需進一步釐清。

**若你是外部評分者**：建議從最不確定的維度開始——通常是「可再利用性」（混合授權易引起判斷分歧）與「技術門檻」（對「複雜 API 請求」的認知有經驗差異）。
